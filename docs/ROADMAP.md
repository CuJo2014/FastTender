### Расширения сверх изначального плана (накопились в ходе пилота)

Решали реальные кейсы заказчика — оказались критичными для usability:

| Фича | Зачем | PR/коммит |
|---|---|---|
| **Item.code_1c** отдельно от Артикула | 1С хранит два разных идентификатора | migr 0005 |
| **Уникальность артикула per-brand** | Один артикул у разных производителей = разные товары | migr 0006 |
| **Supplier.prefix + Item.supplier_sku** | Внутренние стабильные SKU для прайс-позиций | migr 0007 |
| **Item.linked_catalog_item_id** | Связка прайс↔каталог (auto + manual) | migr 0008 |
| **Supplier transformations** (brand_regex, vat, defaults, force_manufacturer) | Разнобой формата прайсов без кастомных парсеров | meta.transformations |
| **Parser: exclude_fields** | CODE_1C не должен искаться в прайсах поставщиков | header_detector |
| **Parser: negative-list** | ТНВЭД/штрих-код не путать с code_1c | header_detector |
| **External backup → Google Drive** | Защита от потери данных | `deploy/backup-cron.sh` |
| **Disaster Recovery план** | Восстановление за ~60 минут | `deploy/RECOVERY.md` |

## Фаза 2 — MVP (после успешной приёмки прототипа)

Целевая длительность: 8-12 недель. Полный план — раздел 17 архитектуры.

**Топ-5 главных тем:**

1. **Векторный/семантический поиск** (pgvector + sentence-transformers)
   — закроет «синонимические» совпадения которые сейчас провисают
2. **OpenSearch как замена pg_trgm** — лучше масштабируется, поддерживает
   множественные индексы
3. **Извлечение атрибутов** (раздел 4.2) — character split «Болт М10х40 DIN933»
   → `{диаметр: 10, длина: 40, стандарт: DIN933}` через regex/NER
4. **LLM-валидация неоднозначных кейсов** (раздел 4.7) — когда матчер
   неуверен, спросить LLM
5. **RBAC + Подразделения** (см. ниже) — отдельная подтема

**Параллельно:**
- PDF/DOCX/OCR (раздел 10.1)
- Веб-парсинг прайсов с сайтов (раздел 11)
- Расширенная аналитика, дашборды
- Генерация КП (DOCX/PDF) по шаблонам клиента
- Multi-tenancy (раздел 5.5) — несколько компаний на одной установке
- Интеграции с 1С/ERP/e-mail

### RBAC + Подразделения (зафиксировано 1 июня 2026)

Заказчик обозначил требование разграничения доступа:

1. **Роли:** отдельные права на загрузку каталога, прайсов, спецификаций.
2. **Видимость спецификаций по подразделениям:** у пользователя
   ограниченный список подразделений, видит только «свои» спеки.

**Что нужно проработать на дизайн-фазе:**

| Вопрос | Зачем |
|---|---|
| Какие конкретно роли нужны (catalog_admin / pricelist_admin / specification_user / viewer / admin)? | Определяет матрицу прав на endpoints |
| Что с верификацией — отдельная роль или часть specification_user? | UX-вопрос |
| Откуда список подразделений — импорт из 1С/AD/HR или ручное создание в системе? | Влияет на интеграцию |
| Один user → одно подразделение или несколько? | Влияет на модель данных (FK vs M2M) |
| Каталог и прайсы — глобальные или скоупятся по подразделению? | Влияет на row-level security |
| Иерархия подразделений (отдел → группа) или плоский список? | Влияет на матчинг прав |
| SSO (Google Workspace / Active Directory / Keycloak) или собственная auth? | Сокращает время разработки если есть готовый IdP |
| Видит ли руководитель спеки подчинённых? | Влияет на role/department дизайн |
| Может ли user «забрать» спеку другого (при отпуске/болезни)? | Нужен ли admin override |
| Audit log — что писать (входы, создание, удаление, верификация)? | Соответствует разделу 13.2 |

**Объём работ (предварительная оценка):** 2-3 недели разработки если
все вопросы выше зафиксированы заранее. Включает: модели User/Department,
JWT-аутентификация, role/department middleware, UI логина и
пользователей, row-level фильтр на Specification, audit log.

**Текущее состояние:** Basic Auth через Caddy, один пароль на всех.
Достаточно для пилота с одним менеджером.
