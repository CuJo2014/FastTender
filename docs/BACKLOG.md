Inner `CandidatesTable` не трогать. Без новых npm-зависимостей.
  - **Ключевать ширины по стабильному `columnId`, НЕ по индексу/позиции** (решено 2026-06-06). Это закладка под будущие кастомные колонки на покупателя (Покупатель А → «НомерЕНС», Покупатель Б → «Точка доставки»): глобальная мапа `{columnId: width}` тогда устойчива к разным наборам колонок у разных спек — каждая колонка получает свою ширину, не конфликтуя. Если ключевать по индексу — наборы колонок «съедут» друг на друга.
  - **Ширина — преференс пользователя, а не покупателя.** Поэтому НЕ хранить «на покупателя»: два менеджера на одной спеке могут хотеть разную ширину. Сейчас юзер-скоуп = localStorage; при RBAC (Фаза 2) переедет в профиль пользователя.
  - **Кастомные колонки на покупателя — отдельная фича Фазы 2** (дизайн-фаза, в пакете с RBAC + Подразделениями; это изменение модели данных + экспорта, не косметика). Спешки нет: данные уже не теряются — `SpecItem.raw_row` (JSONB) хранит все сырые колонки исходного файла, а `Client.meta` (JSONB) — естественное место под конфиг «какие колонки показывать этому покупателю».
