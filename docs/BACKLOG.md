### Матчинг

#### Шумный запрос: клиент свалил ТЗ в «наименование» (диагностика 2026-06-16) — план
Ветка `feature/spec-matching-denoise`.

**Симптом.** Прод-спека `0616_003.xlsx` (`b49c7d7f-636d-4151-a9ad-95ed0ea97a44`):
клиент вписал в одну ячейку наименования целое ТЗ — основное имя товара
(«АППАРАТ ИНВЕРТОРНЫЙ КЕДР MULTIARC-2000», «Электроды Ø2,5 ОК 53.70 ESAB»)
плюс ~450 символов про комплектацию, опции, аттестат НАКС, РД 03-614-03, ГОСТ,
упаковку. Артикул/производитель/характеристики в отдельных колонках пусты — всё
в `name_raw`. Нужные позиции **есть в каталоге**, но топятся шумом → менеджер
искал руками.

**Причина** (`pg_trgm.py:search_lexical`, скор = `GREATEST(ts_rank, similarity(name, весь_запрос))`):
1. Триграмм-similarity по блобу ≈0 — каталожное имя короткое, совпадает малая доля
   500-символьного запроса.
2. `ts_rank` размывается и тянет чужие позиции по шумовым токенам (генератор,
   кабель, вилка, 220в, гост…). В проде в топ-1 вышли «щётка для пола» (conf 0.06)
   на сварочник и «блок автозапуска генератора»/«мундштук газового резака»
   (conf 0.05) на электроды ESAB.
3. Где текст короче (стр. «УОНИИ 13/55»), верные электроды нашлись на 0.44–0.46,
   но ниже порога 0.5 → «Не найдено».

Это усиленный вариант болезни name-only (чинили 2026-06-05): без артикула потолок
≈`w_lexical`, а шум добивает скор до нуля.

**Подтверждение на проде** (обрезанный запрос против блоба):
- электроды ОК 53.70: блоб → мусор 0.05; «электроды ок 53.70 esab 2,5» → **0.794**
  (верный товар топ-1);
- КЕДР MULTIARC-2000: блоб → «щётка» 0.06; обрезка → 0.59, в каталоге реально есть
  «сварочный инвертор (200а) multiarc-2000 кедр».

**План (по приоритету):**
- **A. Денойз-запрос при парсинге (первый шаг).** Из `name_raw` строить отдельное
  `search_text`: «голова» наименования до первого boilerplate-маркера, хвост
  отбрасываем. Маркеры: `комплектация`, `опция`, `аттестат`, `накс`, `соответстви`,
  `требовани`, `гост`, `рд `, `ту `, `упаковка`, `напряжение сети`, `температура`,
  `габариты`, `исполнение`, `категория помещения` и т.п. Полный текст оставить в
  `name_raw` (отображение + извлечение кодов `extract_*` идёт по полному).
  Лексический поиск (`search_lexical`) вести по `search_text`.
  ➕ макс. эффект за минимум усилий (0.05→0.79), детерминированно, без LLM и без
  вопроса суверенитета данных. ➖ эвристика; список маркеров подбирать; мерять на gold.
- **B. Устойчивый к длине лексический скоринг.** В `search_lexical`: при длинном
  запросе не считать `similarity(name, весь_запрос)`; обрезать запрос для триграмм;
  нормировать ранг по доле совпавшего ДОКУМЕНТА, не запроса; рассмотреть `ts_rank_cd`.
  Хорошо комбинируется с A.
- **C. Извлечение ключевых терминов** (TF-IDF/стоп-словарь + веса; редким моделям/
  бренду больший вес) — принципиальная версия A, обобщается лучше. Больше работы.
- **D. LLM-нормализация строки** (Фаза 2): Haiku → `{наименование, бренд, модель,
  атрибуты}`. Робастно, но стоимость/инфра + суверенитет данных (BYOK/L3).
- **E. Ручной поиск в строке** (полезно всегда, дёшево): поле «искать как…» +
  «↻ повторный матчинг по строке». Формализует текущий ручной обход.
- **НЕ делать в одиночку:** просто снижать порог 0.5 — верный товар тут даже не
  топ-1, показ кандидатов ниже порога не поможет.

**Рекомендация:** A + замер на gold dataset, при необходимости B; E как
UI-предохранитель; D — Фаза 2.

**Нюансы:** эффект только на НОВЫЕ прогоны → эту и старые спеки перематчить
(«↻ Повторный матчинг»/перезалив). Перед внедрением посеять строки `0616_003`
в gold dataset, замерить recall@5 до/после.

- **Тонкая настройка весов** в reranker (`w_article=0.5, w_lexical=0.5, boost_brand=0.1, boost_unit=0.05`) — должно подстраиваться под gold dataset когда заказчик его заполнит
- **Semantic search через pgvector** — Phase 2 (`PgVectorRepository` — заглушка)
- **Атрибуты товара** (Phase 2) — сейчас `attributes={}` всегда
